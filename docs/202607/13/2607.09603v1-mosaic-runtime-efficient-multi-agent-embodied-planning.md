## 摘要
基于LLM的多智能体具身规划由于执行延迟过高而仍然不实用。我们将失败动作识别为主要瓶颈，其源于两个核心挑战：部分可观测性下的不准确状态跟踪，以及产生冗余或冲突动作的低效协调。我们提出了Mosaic，一个运行时高效的多智能体规划框架，解决了这两个挑战。Mosaic通过以智能体为中心的语义记忆维护准确且轻量级的状态跟踪，该记忆以相对坐标存储对象，支持几何变换和协调。它通过整数线性规划在每一步规划中分配动作，确保物理可行性以及智能体间的协调约束，从而实现高效协调。在AI2-THOR和搜索救援基准测试中，Mosaic实现了27-32%的执行加速，30-33%更少的LLM调用，25-31%更少的步骤，以及4-10%更高的成功率。这些结果表明，高效的记忆和约束引导的协调对于可扩展、低延迟的多智能体规划至关重要。

## Abstract
LLM-based multi-agent embodied planning remains impractical due to prohibitively high execution latency. We identify failed actions as the dominant bottleneck, stemming from two core challenges: inaccurate state tracking under partial observability and inefficient coordination that produces redundant or conflicting actions. We introduce Mosaic, a runtime-efficient multi-agent planning framework that addresses both challenges. Mosaic maintains accurate yet lightweight state tracking through agent-centric semantic memory that stores objects in relative coordinates, enabling geometric transformations and coordination. It ensures efficient coordination through Integer Linear Programming that allocates actions at every planning step, enforcing physical feasibility and inter-agent coordination constraints. Across AI2-THOR and search-and-rescue benchmarks, Mosaic achieves 27-32% faster execution, 30-33% fewer LLM calls, 25-31% fewer steps, and 4-10% points higher success rates. These results demonstrate that efficient memory and constraint-guided coordination are critical for scalable, low-latency multi-agent planning.

### 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：通过两个关键设计实现运行时高效的规划：1）轻量级且准确的状态记忆；2）基于约束的动作选择来避免失败。
- **算法流程**：
    - **智能体中心语义记忆 (Agent-centric Semantic Memory, ASM)**：
        - **技术细节**：每个智能体维护一个结构化的记忆，记录所有任务相关物体和目的地的 **相对坐标**（角度和距离，例如物体A在右边4步远）。
        - **关键机制**：
            1.  **轻量级更新**：当智能体移动时，可以根据自己的运动轨迹（位移和转向）自动更新记忆中所有物体的相对位置，无需重新探索或构建全局地图。
            2.  **智能体间信息转移**：当一个智能体发现物体后，可以通过几何变换，将该物体的相对坐标信息传递给其他智能体，实现信息共享。
            3.  **语义化接口**：为LLM的推理，将连续坐标映射为离散的语义描述（如“就在前方”，“很远”），简化LLM的推理负担。
    - **基于约束优化的动作选择 (Action Selection via Constraint Optimization)**：
        - **技术细节**：在每个规划步，引入一个 **整数线性规划（ILP）框架**，来解决LLM提出的动作候选与跨智能体执行可行性之间的矛盾。
        - **关键机制**：
            1.  **LLM生成候选动作**：LLM为每个智能体产生少量（例如3个）候选动作。
            2.  **ILP约束求解**：ILP框架对这些候选动作进行全局选择，确保最终选出的**联合动作**满足一系列 **硬约束**：
                -   **资格约束**：动作必须满足前提条件（如手空了才能拿东西）。
                -   **碰撞/干扰避免**：阻止两个智能体走到同一位置或操作同一物体。
                -   **资源排他性**：确保单个资源（如工具）不会被多个智能体同时使用。
                -   **联合行动**：确保需要多人协作的任务（如搬重物）有足够的人手。
            3.  **代价函数优化**：在满足硬约束后，ILP通过**代价函数**选择效率最高的组合。该代价函数惩罚**低效模式**和**负载不均衡**：
                -   **低效模式**：重复循环、持续失败、来回震荡、原地停滞等。
                -   **负载不均衡**：惩罚当前相对工作量较少的智能体选择“空闲”动作，鼓励所有智能体都积极工作。

- **公式要点**：
    - 目标函数：最小化总代价 J，包括时空惩罚 C_penal 和负载惩罚 C_load。
    - 约束条件：由一个决策变量 z 来控制每个智能体在每个规划步选择哪一个候选动作。

3.  **无多模态融合**：系统完全依赖文本描述（语义记忆和观察文本）进行推理，并未利用原始的视觉/激光雷达流进行更底层的感知、定位和避障。这可能限制了在更复杂物理环境中的应用。
4.  **依赖LLM能力**：虽然Mosaic减少了对LLM的依赖次数，但核心的候选动作生成仍高度依赖LLM。如果LLM本身质量不高或无法生成有价值的候选动作，ILP的效果会非常有限。
5.  **手动设计的约束**：ILP中的约束和代价函数需要人工设计和调参（表8）。对于新的、未预见的场景，可能需要重新设计约束，通用性存在一定的天花板。
