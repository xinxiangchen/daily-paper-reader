## 摘要
从自然语言合成可部署的机器人行为树需要约束保证，以确保每个生成的行为树仅引用机器人实际可执行的技能。现有的基于LLM的行为树合成方法通常将这一约束责任交给提示编写者。当编写者不知道机器人可执行哪些技能、这些技能如何参数化或机器人运行时软件如何约束有效行为树结构时，这种部署就会变得脆弱。本文提出了一种合同约束的行为树合成架构，其中编码代理在合成行为树进行验证和执行之前，查询机器人端的模型上下文协议服务器以获取显式合同，该合同包含技能库、允许的行为树操作符以及可选的行为树组合模板。在我们的框架中，非专业操作员发出自然语言命令而无需了解机器人实现细节，同时机器人运行时验证门在执行前强制检查正确性。我们在PyRoboSim中的110个模拟任务和一个物理Husarion Panther机器人的14个任务上评估了两个LLM：一个闭源模型（Sonnet 4.6）和一个较小的开源模型（Gemma4:31b）。结果表明，合同约束实现了近乎完美的行为树验证和高任务成功率，行为树组合模板显著恢复了较小模型在反应式控制流任务上的成功率，并且该架构可迁移到运行对操作员和代理均不透明的Nav2栈的物理硬件上。

## Abstract
Synthesizing deployable robot behavior trees (BTs) from natural language (NL) requires grounding to ensure every generated BT references only skills a robot can actually execute. Existing LLM-based BT synthesis approaches often place this grounding responsibility on the prompt author. This makes deployment brittle when the author does not know which skills the robot can execute, how those skills are parameterized, or how the robot runtime software constrains valid BT structure. This paper proposes a contract-grounded BT synthesis architecture in which a coding agent queries a robot-side Model Context Protocol (MCP) server to retrieve an explicit contract consisting of a skill library, permitted BT operators, and optional BT composition templates, before synthesizing a BT for validation and execution. In our framework, non-expert operators issue NL commands without knowledge of robot implementation details, while a robot runtime validation gate enforces correctness before execution. We evaluate two LLMs, a closed model (Sonnet 4.6) and a smaller open-source model (Gemma4:31b), across 110 simulated tasks in PyRoboSim and 14 tasks on a physical Husarion Panther robot. Results show that contract grounding enables near-perfect BT validation and high task success, that BT composition templates substantially recover success on reactive control-flow tasks for the smaller model, and that the architecture transfers to physical hardware running a Nav2 stack opaque to both operator and agent.

## 1. 核心问题与整体含义（研究动机和背景）
- **研究动机**：机器人行为树（BT）是一种可解释、模块化的控制架构，但手动构建需要机器人技能、参数接口和树结构等专业知识，非专业操作员难以掌握。现有基于LLM的BT合成方法将“接地”责任推给提示编写者，导致部署脆弱：当编写者不知道机器人实际可执行技能、参数化方式或运行时软件约束时，生成的BT容易无效。
- **整体含义**：本文提出一种**合同约束（contract-grounded）的BT合成架构**，通过编码代理（coding agent）主动查询机器人端暴露的显式合同（技能库、允许的BT操作符、组合模板），再合成BT，使得非专家用户只需自然语言指令，不需要了解机器人实现细节。机器人运行时验证门（validation gate）在执行前强制检查正确性。

## 2. 方法论：核心思想、关键技术细节
- **核心思想**：将接地从提示作者转移到机器人系统。机器人通过MCP服务器暴露一个机器可读的合同 \(C = (K, O, R)\)，其中 \(K\) 为技能库（含名称、参数模式、语义），\(O\) 为允许的BT操作符集，\(R\) 为可选的BT根茎（rootstock，即模板化的BT片段，用于编码常见任务结构）。编码代理在合成BT前先查询合同，然后根据合同约束生成BT，提交给机器人验证门（检查BT语法、技能合法性、参数类型），验证通过后执行。如果失败，返回结构化错误，代理可重试（最多5次）。
- **关键技术细节**：
  - **MCP服务器工具**：暴露5个工具（get_skill_library, get_bt_operators, get_rootstocks, get_world_vocabulary, send_to_robot），代理通过调用这些工具获取合同信息并提交BT。
  - **验证门**：检查BT是否解析为有效树，只使用允许的操作符，所有叶节点为合同中的技能且参数类型兼容。
  - **合成方式**：每个任务在一个独立的Claude Code子进程中完成，系统提示告诉代理查询MCP服务器，然后构建BT并提交。
  - **两种模型**：Sonnet 4.6（闭源、大规模、API访问）和 Gemma4:31b（开源、小模型（31B）、本地部署）。

1. **合同接地实现近乎完美的BT验证**（Q1）：Sonnet在M-Core下Valid@1达到100%（Core60和Lang50），B1为98%；Gemma的Valid@1较低（85-92%），但M-Core下成功率达到83%和82%，相比B1的50%大幅提升。

3. **架构可迁移到物理硬件**（Q3）：在Panther机器人上，两种模型均能生成有效BT，且B1与M-Core性能接近（可能因物理根茎收益较小）。
4. **合同外请求的拒绝率中等**：OOC10中Sonnet拒绝7/10，Gemma 5/10（仿真）和7/10（物理）。部分失败表现为近似BT或角色外响应，说明合同本身不足以保证完美拒绝。

## 7. 优点
- **方法创新**：将“接地”从提示编写者转移到机器人端合同服务器，通过MCP协议实现标准化接口，降低了非专家使用门槛，并增强了对机器人运行时变化的鲁棒性。
- **验证门机制**：在BT执行前进行合同约束检查，防止无效BT执行，提高了安全性。
- **根茎模板**：解决了小模型难以构造复杂反应式控制流的问题，是可迁移的知识表示。
- **实验全面**：涵盖仿真与物理硬件、多种任务原型、语言变体、两个不同规模的LLM、消融分析、失败分类，结果具有说服力。
- **实用价值**：物理机器人实验证明了在预编译、不透明的ROS2/Nav2栈上的可行性，生成时间（预热后5-10秒）适合实时部署。

## 8. 不足与局限
- **合同质量依赖**：合成效果取决于所暴露合同的质量和详细程度，特别是技能模式描述。若合同不准确，合成可能失败。论文未讨论合同生成或维护的自动化。

- **物理任务成功判定主观**：物理任务成功由人工观测，可能引入偏差；仿真严格但环境有限。
- **单任务、无在线重规划**：目前是单次合成完整BT，不支持执行中在线重规划或与用户交互，限制了动态场景适用性。

- **资源消耗高**：Gemma本地需4块高端GPU，Sonnet依赖商业API，对资源有限团队可能不友好。
